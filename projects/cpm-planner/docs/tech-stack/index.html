2. Frontend Decisions & Justifications​

Next.js 16 (App Router) & React 19​

  • Why: Next.js App Router provides hybrid rendering where public landing and auth pages are pre-rendered statically on the server, while project dashboards leverage client-side reactivity. React 19's async transitions simplify UI updates.

React Flow (@xyflow/react v12)​

  • Why: Visualizing a network of dependencies is the core requirement of CPM. React Flow is the leading library for node-graph visualization, supporting panning, zooming, custom node containers, draggable coordinates, and edge markers out of the box.

Tailwind CSS (v4)​

  • Why: Tailwind CSS v4 provides a fast, utility-first styling pipeline that integrates with custom CSS variables. It enables the implementation of the Notion-inspired design system tokens (such as soft canvases, rounded buttons, and sticker accent cards) without writing verbose CSS class sheets.

Zustand (v5)​

  • Why: Zustand provides an extremely lightweight, hook-based global state manager. It is used here to share transient workspace states (such as active projects, open sidebar panels, and user context) across nested React Flow nodes and layout bars without context-re-render performance penalties.

TanStack React Query (v5)​

  • Why: Scheduling calculations frequently invalidate data. React Query manages request deduplication, loading states, error boundaries, and cache invalidation policies, guaranteeing that when the CPM engine updates a project, all UI tables and charts update automatically.

3. Backend & Native Computation Justifications​

Custom Node.js Web Server (server.mjs) & WebSocket Server (ws-server.mjs)​

  • Why: Standard serverless functions struggle to maintain persistent WebSocket connections. Running a unified Node.js server via server.mjs enables hosting the REST API, serving Next.js pages, and holding open WebSocket connections on the same port, simplifying containerization.

Compiled C++17 Engine (cpm_cli)​

  • Why: Scheduling algorithms (Topological Sorts, DFS, Forward/Backward Date passes) are CPU-bound and become highly inefficient in single-threaded JavaScript runtimes when scheduling graphs scale to thousands of nodes. A compiled C++ executable processes heavy graph operations in microseconds and is launched on-demand to safeguard Web server event loops from blocking.

4. Persistence & Caching Rationale​

PostgreSQL & Prisma ORM​

  • Why: CPM schedules have highly relational structures (Tasks belong to Projects; Dependencies map tasks to other tasks; Members belong to Departments). PostgreSQL is a robust, ACID-compliant relational database. Prisma provides type-safe query building and auto-generates schema migrations.

Redis (ioredis with Mock Fallback)​

  • Why: Multiple users viewing a project triggers high-frequency reads. Caching statistics, member lists, and critical paths in Redis avoids hitting the database on every route request. The custom cache wrapper falls back to an in-memory map if Redis goes offline, ensuring the application remains functional in minimalist dev environments.

5. AI / Automation & Simulation Technologies​

Autopilot Scheduling Modes​

  • Why: The system supports Manual, Assisted, Semi-Automatic, and Full Autopilot modes. This AI-driven proactive orchestration uses task analytics and risk scanning to suggest or automatically make scheduling adjustments (e.g. rescheduling tasks when a deadline-miss is predicted).

Synthetic Persona & Classifier Engine​

  • Why: To train and evaluate the deadline-miss prediction model, the repository includes a synthetic dataset generator (generate_dataset.py inside scripts) that models user productivity profiles ("College Student," "Freelancer," "Working Professional," "Entrepreneur"). It uses sequential LLM labeling pipelines with fault-tolerant backoff retry schedules to label productivity logs and generate realistic schedules.
