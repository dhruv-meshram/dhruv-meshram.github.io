Freshwash DBMS

A high-performance sharded database engine featuring a custom B+ Tree storage engine, ACID-compliant write-ahead logging (WAL), horizontal sharding middleware, and a React management dashboard.

Overview & Problem

Traditional relational database management systems introduce significant overhead for simple read/write operations due to comprehensive SQL parsers, query planners, and heavy execution engines. Freshwash DBMS was designed as a lightweight, specialized alternative to achieve maximal insert and query speeds for high-volume transactions while preserving the safety guarantees of traditional database engines.

The project builds a complete database system from scratch. By writing a custom B+ Tree storage engine in C++ and distributing data transparently across shard nodes using a Python routing middleware, we created an engine capable of handling hundreds of thousands of transactions per second with horizontal scalability.

System Architecture

The database architecture is split into three core layers: the C++ Storage Nodes, the Python Sharding Middleware, and the React Admin Portal.

C++ B+ Tree Engine: An on-disk index manager with fixed-size pages. Nodes are split and merged dynamically to ensure O(log N) index lookups. Pointer structures are serialized directly to file offsets using a custom page buffer pool.
ACID & WAL: Write-Ahead Logging writes transaction payloads to disk sequentially before updating the B+ tree. In case of unexpected server crashes, a recovery parser replays the WAL to ensure transaction durability (Redo) and rolls back partial writes (Undo).
Sharding Middleware: Employs consistent hashing algorithms in a Python network wrapper to distribute primary keys across distinct physical C++ storage nodes, minimizing re-sharding overhead.

Challenges & Key Learnings

The primary technical challenge lay in memory serialization. Because B+ Trees are traditionally pointer-heavy, translating native C++ pointers to stable disk-based representations required building a virtual offset architecture. Every page node refers to children by virtual Page IDs rather than memory addresses, allowing the Buffer Manager to swap pages in and out of memory seamlessly.

Additionally, load testing using Locust highlighted potential race conditions during high-concurrency inserts. This led to the implementation of fine-grained latching (latch crabbing) on B+ Tree nodes, which allows multiple threads to traverse the index tree concurrently without compromising structure integrity.
